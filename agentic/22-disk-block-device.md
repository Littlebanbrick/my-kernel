# 22 — 磁盘:第一个"块设备"

## 起点:磁盘和内存的根本区别

之前做过的所有模块——printf、调度、分页——都是对**内存**操作。内存有一个我们习以为常的性质:**字节寻址(byte-addressable)**。`*(char*)0x1000` 直接取一个字节,`*(int*)0x2000` 直接取四个字节,地址就是地址。

磁盘打破了这个前提。这次第一次碰上一个**不听话的设备**:它有机械动作、有自己的节奏,你不能"读地址 X"就拿到东西,必须用一套约定协议跟它说话,然后等它慢慢把数据吐给你。

所以这篇笔记的核心其实不是 ATA 那几个端口怎么写,而是先立起一个心智模型:

> **内存是"字节设备":每个字节都有地址,直接读。磁盘是"块设备":你能索取的最小单位是一个"块"=一个扇区=512 字节。你说不出"给我第 73 位",只能说"给我第 5 个扇区"——拿到整块 512 字节后,再在内存里用位运算挑出第 73 位。**

这个区别贯穿后续所有 I/O 设计。先记住它。

---

## 块设备 vs 字符设备(本次最重要的辨析)

Linux 内核把外设分成两大类。这分类不是随便分的,它**直接决定了驱动该怎么写**:

|  | 字符设备 (char device) | 块设备 (block device) |
|---|---|---|
| 例子 | 键盘、串口、鼠标 | 磁盘、SSD、U 盘、光盘 |
| 读写粒度 | 字节流,一个一个来 | 整块,512 的倍数 |
| 有没有"地址" | 没有,顺序读 | 有,LBA 编号 |
| 数据什么时候来 | 不确定(异步)→ 中断 | 你主动要(同步)→ 轮询/DMA |
| 本项目里的实例 | 键盘 `kbd.c` | 磁盘 `ata.c` ← 新 |

辨析这两类,关键看一件事:**"我知不知道数据什么时候来"**。

- **键盘是字符设备**。你不知道用户什么时候敲、敲什么。所以必须靠**中断**:它一敲就来打断 CPU,CPU 去把那个字节拿走。这是异步的。
- **磁盘是块设备**。是**你主动**要读的——你点名要"第 5 扇区"。所以你**知道**现在就在等它,与其设中断等它回来打断你,不如直接原地转圈问"好了吗",还省了中断切换的开销。这是同步的。

这就是为什么 `kbd.c` 用中断、`ata.c` 用轮询(polling)——**不是随意的实现选择,是设备性质决定的**。字符设备天生异步,块设备天生同步。

> 一个推论:块设备其实也能用 DMA(让磁盘自己把数据写进内存,写完发一个中断通知 CPU),那样 CPU 就不用轮询了。真磁盘都这么干。但我们为了简单选了 PIO——CPU 亲自一个字一个字搬。慢,但零配置、绝对正确。教学内核的正确取舍。

## 扇区:为什么是 512 字节

真实磁盘是一片(或几片叠起来的)旋转的**盘片**。一个**磁头**悬浮在盘片上方,能**沿半径方向移动**到任意一圈,但**不能跳着读**。盘片转起来,磁头扫过一个圆环——这个圆环叫**磁道 (track)**。

但你不可能"读一整圈",那一圈数据太多。所以把每个磁道切成一段段等长的弧,每段 **512 字节**,叫**扇区 (sector)**:

```
            ┌──── 磁头沿半径移动(选哪一圈磁道)
            │
        ╭───┴────────────────────╮
       ╱  ┌─扇区─┐ ┌────┐ ┌────┐  ╲     盘片在转 →
      │  │ 512B  │ │512B│ │512B│   │
      │   ╰──────╯  ╰────╯  ╰────╯  │
       ╲      ← 整圈是一个磁道      ╱
        ╰────────────────────────╯
              ↑ 盘片在旋转
```

所以读一个扇区的**物理过程**是三步:

1. 磁头**移到**目标磁道所在的半径;
2. **等**盘片转,转到目标扇区正好转到磁头底下;
3. 磁头把那一整段 512 字节**扫**进控制器,控制器再从数据端口吐给 CPU。

第 2 步那个"等转过来"就是磁盘慢的**物理根源**——机械动作是毫秒级,CPU 是纳秒级,差六个数量级。这就是 `ata.c` 里 `ata_wait_not_busy` / `ata_wait_drq` 那两个自旋循环存在的根本原因:CPU 在等一个慢吞吞的机械动作,只能干转圈问"好了吗?"。

> **512 这个数字是历史遗物**。1980 年代 IBM PC/AT 的扇区大小就是 512,后来变成事实标准沿用至今。现代磁盘物理扇区其实多是 4096 字节(4K),但为了兼容老软件,对外仍假装成 512(这叫 512e)。我们这里不用管这个细节,磁盘对我们就是"512 字节一块"。

## CHS → LBA:从三维坐标到线性编号

历史上给扇区编号用的是它**真实的几何坐标**,叫 **CHS**:

- **C**ylinder(柱面)= 哪一圈磁道(半径位置)
- **H**ead(磁头)= 哪一片盘片、哪一面
- **S**ector(扇区)= 这一圈上的第几段(角度位置)

所以一个扇区的"真地址"是个**三维坐标** `(C, H, S)`。老软件得查磁盘的几何表(每圈多少扇区、共多少磁头……)自己算,痛苦且因盘而异。

**LBA(Logical Block Addressing)就是这个痛苦的解药**:把所有圆环、盘面、角度**假装拆开摊平成一条直线**,从 0 开始编号:

```
   CHS 真实几何(一圈圈、一片片、一段段):
        圈0面0: [扇0][扇1][扇2]...
        圈0面1: [扇0][扇1][扇2]...
        圈1面0: ...
              ↓ "假装摊平"
   LBA 逻辑线性:
        扇区0  扇区1  扇区2  扇区3  扇区4 ...
        (=圈0面0扇0)(圈0面0扇1)(圈0面0扇2)(圈0面1扇0)...
```

你要第 5 个扇区?说 `LBA 5` 就行,**不需要知道它在哪个圈哪个面哪个角度**。磁盘控制器内部替你做 `LBA → CHS` 的换算。软件从此跟物理几何彻底解耦。

**这个抽象和分页是同一类思想,值得点破**:

| | 分页(paging) | LBA |
|---|---|---|
| 真实情况 | 物理内存散落在各处物理页 | 磁盘是圆环+盘面+角度的几何体 |
| 抽象后 | 一条平坦的线性虚拟地址空间 | 一条平坦的扇区编号序列 |
| 谁做翻译 | MMU + 页表(CPU 硬件) | 磁盘控制器 |
| 好处 | 软件不用关心物理内存布局 | 软件不用关心磁盘几何 |

**LBA 之于磁盘 ≈ 虚拟地址之于内存。** 同一套"用线性抽象掩盖物理复杂"的思想,这是你第二次见它了。第一次见是分页,第二次是 LBA——这套思想在 Linux 内核里到处都是。

LBA28:28 位编号 → 最多 2²⁸ = 2.68 亿个扇区 × 512 = **128 GiB**。这就是 `ata.h` 里"up to 128 GiB"的来历。超过就得用 LBA48(用 48 位编号,够 128 PB)。

## ATA PIO 读协议:7 个端口的对话

跟磁盘对话靠 **I/O 端口**。端口就是一个编号,`outb(端口号, 值)` 是往里塞东西(发消息),`inb(端口号)` 是读出来(收消息)。磁盘特别之处:**它有 7 个端口,每个端口各管一件事**:

| 端口 | 名字 | 干什么 |
|---|---|---|
| 0x1F0 | data | **数据门**——512 字节都从这里进出(16 位宽) |
| 0x1F1 | error | 出错时读这里看错误码 |
| 0x1F2 | sector count | "我要读几个扇区" |
| 0x1F3 | LBA low | LBA 的第 0-7 位 |
| 0x1F4 | LBA mid | LBA 的第 8-15 位 |
| 0x1F5 | LBA high | LBA 的第 16-23 位 |
| 0x1F6 | drive/head | LBA 第 24-27 位 **+** 主/从盘 **+** "用 LBA 模式" |
| 0x1F7 | status/cmd | **读** = 查状态;**写** = 发命令 |

两个要特别拎出来的认知:

**① LBA 28 位怎么塞进 8 位端口:拆。**
一个端口 8 位,LBA 是 28 位,塞不下,所以拆成 3+1 份:

```
 LBA (28位):  [27 26 25 24] [23..16] [15..8] [7..0]
                 ↓↓↓↓          ↓        ↓       ↓
              0x1F6低4位     0x1F5   0x1F4   0x1F3
```

`ata.c` 里那些 `lba >> 8`、`lba >> 16`、`lba >> 24` 的位移,干的就是"把一个数拆成几段塞进不同的门"这件事。位移不是炫技,是物理约束逼的。

**② 0x1F6 一个字节塞了 4 件事。**
`0xE0` 这个魔法数字,展开是一个字节 8 位:

```
   0x1F6 字节:
   位:   7   6   5   4 | 3   2   1   0
        ─────────────────┬──────────────
   值:   1   1   1   0 | LBA[27 26 25 24]
         │   │   │   │
         │   │   │   └─ 选盘: 0=主盘, 1=从盘  (bit4=0 才是主盘)
         │   │   └─ 固定为 1(保留位)
         │   └─ 1 = "我用 LBA 模式,不是老式 CHS"
         └─ 固定为 1(保留位)

   固定前缀 = 1110 0000 = 0xE0
```

一个字节,4 件事,全靠位编码。这是**硬件寄存器的常态**:位少、事多,每个 bit 都有专门含义。以后读别的硬件 spec,这种"一个寄存器塞一堆 flag"的设计会反复出现。

**③ 0x1F7 既读又写不冲突。**
同一个端口号,`inb(0x1F7)` 是问它"你忙吗/数据好了吗"(读状态),`outb(0x1F7, 0x20)` 是冲它喊"读!"(写命令)。一个端口两种用法,靠**读还是写**区分,硬件内部按方向接到不同寄存器。

## 握手:三个状态位

控制器状态在 0x1F7(读时是 status):

- **BSY (bit 7)** = 我正忙着,别碰我
- **DRQ (bit 3)** = 数据准备好了,快来拿(Data ReQuest)
- **ERR (bit 0)** = 出错了,放弃

读一个扇区的完整对话就 5 步,全部围绕这三个位:

```
1. 等 BSY=0        ← 控制器闲了才搭理你
2. 往 0x1F6 塞盘号+LBA高4位,往 0x1F2-0x1F5 塞 count + LBA低3字节
3. 往 0x1F7 写 0x20(READ 命令)
4. 等 DRQ=1        ← 一个扇区的 512 字节在数据门后准备好了
5. 从 0x1F0 读 256 个字(16位) = 512 字节,搬进缓冲区
```

多扇区读就是 4-5 步循环:每读完一个扇区,控制器把 DRQ 拉低,凑齐下一个扇区再拉高。所以 `ata_read_sectors` 里那个 `for (i = 0; i < count; i++)` 就是这么来的。

**为什么是 256 次 `inw`?** ATA 总线是 16 位宽(历史),一次传 2 字节。512 ÷ 2 = 256 次。这就是 `ata_read_sector` 里 `u16 *dst` 和 `ATA_SECTOR_SIZE / 2` 的由来,也是当初要在 `utils.h` 补 `inw/outw`(原来只有 8 位的 `inb/outb`,键盘够用、磁盘不够)的原因。

## "读哪"由谁决定:ata_read_sectors 是被动的

这点容易混。`ata_read_sectors(u32 lba, u32 count, void *buf)` 这个函数**根本不知道该读哪**——`lba` 就是个参数,和 `memcpy` 被传一个地址指针一模一样。它是**搬运工**,不是**决策者**。

"读哪个位置"这个决定,全在**调用者**那里:

- 在 `cmd_disk` 里,调用者是**你敲的命令**:`disk 5` → `parse_uint("5")` → `lba = 5`。是**人**决定读第 5 扇区。
- 函数自己没有任何"查找"逻辑,机械地把这个数拆成 28 位塞进端口、发命令、收 512 字节。

那"真正"的操作系统怎么决定读哪个 LBA?**靠文件系统。** 文件系统维护一张表:文件名 → 它的数据散落在哪些 LBA。你要读 `/bin/ls`,文件系统查表说"ls 的数据在 LBA 2000-2050",然后调 `ata_read(2000, 51, buf)`。

```
   应用层:  "我要读 /bin/ls"
                ↓  (文件系统:文件名 → LBA 映射)
   文件系统: "ls 在 LBA 2000-2050"
                ↓  (调用块设备驱动)
   ata.c:    ata_read_sectors(2000, 51, buf)   ← 被动的搬运工
                ↓  (硬件协议)
   磁盘:     吐出 51 个扇区
```

**我们现在还没有文件系统这一层。** 所以决定 LBA 的只有两种来源:

1. **人**(你敲 `disk N`);
2. **写死的位置约定**(比如"内核在 LBA 1")。

没有文件系统,磁盘上"哪个位置放什么"完全由**谁、按什么顺序、写了什么进磁盘**决定——现在这个"谁"是 Makefile 的 `cat boot.bin stage3.pad > disk.img`,它用的是**位置约定**(positional convention):东西在哪,看它在第几个扇区。

## 我们的磁盘长什么样

看 `Makefile`:

```makefile
$(O)disk.img: $(O)boot.bin $(O)stage3.pad
	cat $^ > $@       # disk.img = boot.bin 拼上 stage3.pad
```

`cat` 是按顺序拼接字节,所以:

```
   LBA 0          LBA 1          LBA 2        ...        LBA 80
  ┌──────────┬──────────┬──────────┬─────────┬──────────┐
  │ boot.bin │ stage3 的第0扇 │ 第1扇 │ ...... │ 第79扇   │
  │ (512B)   │ (内核前512字节)│        │         │          │
  └──────────┴──────────┴──────────┴─────────┴──────────┘
   ↑
   512字节正好1扇区(Makefile 强制校验)
```

- **LBA 0 = 你的 boot sector**。开机 BIOS 读它,跳进去。
- **LBA 1 起是你的内核**。stage3.bin 是 objcopy 出来的裸机器码,从 LBA 1 开始一个扇区一个扇区铺。

没有任何文件系统。"LBA 0 是 boot、LBA 1 是内核"不是磁盘告诉你的,是**你的 Makefile 这么拼的**。

## 验证:为什么 55 aa 证明全链路通

`disk 0` 跑出来最后一行:

```
1f0: 00 00 00 00 00 00 00 00 00 00 00 00 00 00 55 aa  ..............U.
```

`1f0:` 这一行显示的是**扇区内偏移 0x1F0 到 0x1FF** 的 16 个字节。`55 aa` 在这行的第 15、16 个位置,即偏移 **0x1FE 和 0x1FF**。

- 一个扇区 512 字节,偏移范围 0 ~ 511(0x1FF)。
- **0x1FE = 510,0x1FF = 511**,正好是扇区最后两个字节。
- PC 规定:**可引导扇区的最后两个字节必须是 `0x55 0xAA`**(引导签名 boot signature)。BIOS 开机就是看这两个字节判断这扇区是不是真 boot sector,是才跳进去。

所以我们读出来的 512 字节,结尾恰好是 `55 aa`,说明 **512 字节一个不差地从磁盘正确读到了内存**。如果只读了 256 字节(漏一半),看不到结尾的签名;如果地址偏了,结尾也不会是 55 aa。这个签名是一个天然的"校验码",看到它 = 端口号对、命令对、LBA 拆分对、256 次 inw 对、缓冲区映射对——全链路通。

## 下一步:exec

磁盘能读了,exec 就有了物质基础。exec 要做的是:

> "我要加载程序镜像 → 程序镜像在磁盘哪个 LBA?→ 没有文件系统,约定一个位置(比如 LBA 90 往后放程序)→ 调 `ata_read_sectors(90, N, buf)` → 跳进去执行。"

`ata_read_sectors` 在那一刻依然只是个**给个号码、吐 512 字节**的笨函数。"读哪"这个智力活,在 exec 那一层。我们这一步把"搬运工"和"决策者"的界限看清了——这正是 exec 该建立的认知。
