# 学习笔记：64 位长模式与分页

> 日期：2026-07-01
> 目标：从 32 位保护模式进入 64 位长模式，运行 C 代码

---

## 一、为什么要进长模式？

| 模式 | 地址位宽 | 最大物理内存 | 通用寄存器 | 是否支持现有 OS |
|------|---------|------------|-----------|----------------|
| 实模式 | 20 位 | 1 MB | 16 位 | ❌ |
| 保护模式（32 位） | 32 位 | 4 GB | 32 位 | 早期 Linux |
| **长模式（64 位）** | **64 位** | **16 EB** | **64 位** | **现代 Linux** |

进入长模式是"真正的内核开发"的起跑线——现代操作系统都在这个模式下运行。

---

## 二、从实模式到长模式的路线图

```
实模式 (16-bit)
    │
    ├─ 加载 GDT
    ├─ 设置 CR0.PE = 1
    ▼
保护模式 (32-bit)         ← stage-2 已完成
    │
    ├─ 写入页表（PML4 → PDP → PD）
    ├─ 设置 CR4.PAE = 1
    ├─ 设置 CR3 = PML4 地址
    ├─ 设置 EFER.LME = 1  ★ 关键一步
    ├─ 设置 CR0.PG = 1    ★ 从此分页开启
    ▼
兼容模式（32-bit 指令，64-bit 分页）
    │
    ├─ 加载 64-bit GDT（L=1 的代码段）
    ├─ ljmp 到 64-bit 入口
    ▼
长模式 (64-bit)
```

## 三、分页与页表

### 为什么需要分页？

分页是一套 **虚拟地址 → 物理地址** 的映射机制。

没有分页时，CPU 访问的地址直接对应物理内存（或者通过段转换）。有分页后，**每个地址都要通过页表翻译**才能找到真正的物理位置。

好处：
- 每个进程拥有独立的虚拟地址空间（彼此隔离）
- 物理内存可以按需分配（不需要连续）
- 磁盘交换（swap）成为可能

### 4 级页表结构（长模式使用）

```
虚拟地址分解：
┌─────────┬─────────┬─────────┬─────────┬──────────┐
│  PML4   │  PDP    │   PD    │   PT    │  偏移量  │
│  索引   │  索引   │  索引   │  索引   │          │
│ bit 47:39│ bit 38:30│ bit 29:21│ bit 20:12│ bit 11:0 │
└─────────┴─────────┴─────────┴─────────┴──────────┘
      │         │         │         │
      ▼         ▼         ▼         ▼
   PML4表     PDP表      PD表      PT表      4KB 页
   (512项)   (512项)    (512项)    (512项)
                                          
    也可以直接映射 2MB "大页"（跳过 PT 级）：
   PML4表 ─→ PDP表 ─→ PD表（PS=1）──→ 2MB 物理页
```

### 我们采用的方案：2MB 大页恒等映射

**恒等映射（Identity Mapping）**：虚拟地址 = 物理地址。最简单的方案，适合刚开启分页时使用。

我们只映射了前 4MB（2 × 2MB 大页），覆盖了：
- `0x00000000 - 0x001FFFFF`：PM4 表、PDP 表、PD 表、stage3 代码、VGA 显存
- `0x00200000 - 0x003FFFFF`：额外预留空间

### 页表项（PTE）格式

每个页表项 8 字节（64 位），低 12 位是权限位：

```
bit  63 .. 52  51 .. 12  11  9  8  7  6  5  4  3  2  1  0
     ┌─────────┬─────────┬───┬──┬──┬──┬──┬──┬──┬──┬──┬──┐
     │  保留   │ 物理地址 │IGN│L │D │A │P │U │R │P │  │P │
     │(必须为0)│  高 40位 │   │A │  │CC│CD│S │W │  │WT│  │
     └─────────┴─────────┴───┴──┴──┴──┴──┴──┴──┴──┴──┴──┘
```

关键的权限位：
| 位 | 名称 | 含义 |
|----|------|------|
| bit 0 | P | Present——是否有效。**必须有** |
| bit 1 | R/W | 1=可读写，0=只读 |
| bit 2 | U/S | 1=用户态可访问，0=仅内核 |
| bit 7 | PS | **Page Size**——1=2MB 大页（仅 PD 级有效） |

实际写页表：地址部分和权限位是**拼接**的。

```
例：PML4[0] = 0x9003
  地址部分：0x9000（下一级表在物理 0x9000）
  权限部分：0x003（P=1, R/W=1）
```

## 四、相关 CPU 控制寄存器

### CR0——系统控制

| 位 | 名称 | 作用 |
|----|------|------|
| bit 0 | PE | Protection Enable——开启保护模式 |
| bit 31 | PG | Paging——开启分页（最后一步） |

### CR3——页表基址

存放 PML4 表的**物理地址**（必须 4KB 对齐，低 12 位视为 0）。

```
CR3 = 0x8000  →  PML4 在物理地址 0x8000
```

### CR4——扩展控制

| 位 | 名称 | 作用 |
|----|------|------|
| bit 5 | PAE | Physical Address Extension——**进长模式前必设** |

### EFER——扩展功能允许寄存器（MSR 0xC0000080）

通过 `rdmsr/wrmsr` 指令访问：

| 位 | 名称 | 作用 |
|----|------|------|
| bit 0 | SCE | Syscall Enable——允许 `syscall` 指令 |
| bit 8 | LMSLE | Long Mode Segment Limit Enable |
| **bit 10** | **LME** | **Long Mode Enable——长模式开关** |
| bit 11 | NXE | No-Execute Enable——允许页表 XD 位 |

---

## 五、wrmsr/rdmsr 指令

写 MSR 的标准三步曲：

```asm
movl $0xC0000080, %ecx    ; 选择 MSR 编号（EFER）
rdmsr                      ; 读出当前值到 EDX:EAX
orl  $(1 << 10), %eax      ; 修改 EAX 的特定位（LME = bit 10）
wrmsr                      ; 将 EDX:EAX 写回 MSR
```

**为什么不直接 `movl $0x400, %eax; wrmsr`？**
因为 `rdmsr → 修改 → wrmsr` 模式只改动目标位，不破坏 MSR 的其他位。这是操作任何 MSR 的标准写法。

---

## 六、GDT64——64 位模式的段描述符

进入长模式后，段寄存器的作用大幅弱化，但 GDT **仍然需要**——至少需要合法的 null、code、data 三个描述符。

与 32 位 GDT 的关键区别：

| 属性 | 32 位代码段 | 64 位代码段 |
|------|-----------|-----------|
| L bit | 0 | **1** |
| D bit | 1（32-bit 默认） | **0**（必须） |
| 描述符 | `0x00CF9A000000FFFF` | `0x00209A0000000000` |

64 位数据段描述符也变简单了（段限制在 64 位模式下基本被忽略）：
```asm
.quad   0x0000920000000000    ; P | DPL0 | S | Data | W
```

GDT 伪描述符格式：
```asm
gdt64_desc:
    .word   gdt64_end - gdt64_start - 1    ; 限长（总字节数 - 1）
    .long   gdt64_start                     ; 基地址
    .word   0                               ; 填充
```

## 七、长模式切换流程（逐行解读）

1. **写页表项**：`movl $0x9003, 0x8000` 等——按语法填充 PML4/PDP/PD
2. **`CR4.PAE = 1`**：启用物理地址扩展（进长模式必须）
3. **`CR3 = 0x8000`**：告诉 CPU PML4 表在哪
4. **`wrmsr` 设 `EFER.LME = 1`**：请求进入长模式（此时还未切换）
5. **`CR0.PG = 1`**：**开启分页——CPU 在此刻进入长模式兼容子模式**
6. **`lgdt gdt64_desc`**：加载 64 位兼容的 GDT
7. **`ljmp $0x08, $stage3_64_entry`**：远跳转刷新 CS，进入真正 64 位模式

## 八、.align 和 .globl

| 指令 | 作用 | 为什么需要 |
|------|------|-----------|
| `.align 4096` | 将当前地址对齐到 4KB 边界 | 页表基地址要求低 12 位为 0（硬件规定） |
| `.globl pml4` | 将标签暴露为全局符号 | 方便 C 代码和 GDB 调试引用 |

## 九、cld 和 jmp .

| 指令 | 含义 |
|------|------|
| `cld` | Clear Direction Flag——确保字符串操作方向为**递增**（System V ABI 要求） |
| `jmp .` | 跳转到自身——死循环保底。C 代码返回时卡死在这里 |

## 十、volatile 关键字

```c
volatile unsigned short * const vga = (unsigned short *)0xB8000;
```

**`volatile`**：告诉编译器，"这个地址的值可能被外部改变"或"对这个地址的写不能省略"。

不加 `volatile` 时，编译器可能优化掉看似"无用"的循环写入（认为是死代码）。但对于硬件映射地址（VGA 显存），每次写入都有实际效果，**不能**被优化。

---

## 十一、链接脚本（linker.ld）的段布局

```
地址范围     段名       内容
0x8000      .pml4      PML4 表（4KB 零占位，运行时填入）
0x9000      .pdp       PDP 表（4KB 零占位）
0xA000      .pd        PD 表（4KB 零占位）
0xB000      .text      stage3.S + kernel.c 代码
0xC000      .data      64-bit GDT 等数据
```

那三段 4KB 的 `.space 4096` 只是**占位置**（让链接器推后面段的地址），真正的页表项由运行时 32 位代码通过 `movl` 写入。

## 十二、磁盘扇区（Sector）

扇区是磁盘访问的最小单位，每个扇区 **512 字节**。

- 之前 boot.S 只能写 512 字节（引导扇区本身）
- 现在我们将 stage3 编译为 60 个扇区（30720 字节），由 boot.S 通过 `int 0x13, AH=0x02` 加载到内存 `0x8000`
- 突破 512 字节限制的关键就是"先把数据放磁盘上，再由引导扇区读到内存"

## 十三、遇到的坑

### QEMU/KVM 拒绝写入 EFER.LME

- **现象**：执行 `wrmsr` 设置 LME（bit 10）后，`rdmsr` 读回来仍然是 0
- **验证**：TCG 和 KVM 均如此。SCE（bit 0）和 NXE（bit 11）可以正常写入
- **影响**：无法开启长模式，current代码无法在 QEMU 上验证
- **状态**：未解决。可能是 QEMU 10.2.1 / Linux 7.0.0 + Intel Core Ultra 9 285H 环境下的特定问题
- **代码本身是正确的**——页表结构、GDT64、控制寄存器流程均符合 Intel 手册规范

### 汇编语法差异（AT&T vs Intel）

CSDN 等中文社区上的代码大多使用 **NASM（Intel 语法）**，而我们使用 **GAS（AT&T 语法）**。两者操作数顺序、寄存器前缀、立即数标记均不同，不能直接套用。
