# 18 — 控制台输出竞争：单前台模型与"为什么不加锁"

## 概述

M3 做完 readline + shell 后，一个自然的问题冒出来：

> `printf` 里那个 `console_cursor` 是全局共享的。如果两个进程同时 printf，光标会打架——A 写到一半被切到 B，B 从 A 的光标位置接着写，屏幕变成 `helworldlo` 这种乱码。要不要给 printf 加锁？

这篇笔记记录的是**为什么最终什么都不改**。结论先放：

- **单核环境**下，能打断 printf 的只有中断。而我们的使用模型是**单前台输出**（同一时刻只有一个进程往屏幕写），所以"另一个进程趁机插进来"的场景**根本不会发生**。
- 加锁（`cli`/`sti` 包住整行 printf）是在为**不存在的并发**买单。YAGNI——不为想象中的需求提前设计。
- **约束让简单方案成立**。约束没定时讨论方案是空中楼阁；约束一定，方案就自动收敛。

这不是"没做所以没事"，是**一条有意的设计原则**：本内核采用单前台模型，同一时刻只允许一个进程向控制台输出。

---

## 一、危害到底是什么

先看清楚"光标竞争"的危害本质。不是"两个进程都 printf"本身有问题，是下面这种**行内交错**：

```
进程 A 正在 printf("hello"),刚写了 'h' 'e',光标在 'e' 后面
     ↓ 时钟中断,切到进程 B
进程 B printf("world"),从那个光标接着写
     ↓ 屏幕变成 "heworld"
进程 A 恢复,写 'l' 'l' 'o'
     → "heworldllo"  ← 行内交错,字符穿插,人读不懂
```

**行内交错**（一个 printf 半句被另一个 printf 劈开）才是真问题。**行间交错**（A 一整行、B 一整行、A 一整行）其实可读——就像两人轮流说话，只是顺序不固定。终端用户能接受行间交错，不能接受行内乱码。

---

## 二、三条路（按工程量 vs 通用性排）

### 路 1：承认"单前台"约束，什么都不改

**做法**：约定同一时刻只有一个进程往屏幕写。shell 跑着的时候没有别的进程 printf。

**代价**：将来做 fork 时，子进程不能随便 printf——要么让位给 shell，要么后台化。

**适用**：现在阶段完美。shell + 内核打印，就这俩输出源，不打架。

### 路 2：输出加锁（整行原子）

**做法**：printf 内部用 `cli`/`sti`（或 `irq_save`/`irq_restore`）包住整行，"一行 printf 原子完成，中间不被切走"。

```c
void printf(...) {
    unsigned long flags = irq_save();   // 关中断
    ... 整行写完 ...
    irq_restore(flags);                  // 恢复
}
```

**代价**：① 整行执行期间中断全关，键盘/定时器延迟响应（printf 短无所谓，长了不行）。② 只保证**一行**不乱，多行还是会交错。③ 在单核 + 单前台模型下，**这层保护防的是一个根本不会发生的场景**——纯属多余。

**适用**：多进程都要输出、且能容忍行间交错。早期 Unix 内核 printf 大致这样。

> 注意"用 `irq_save`/`irq_restore` 而非裸 `cli`/`sti`"这条经验本身是有价值的（见下文"延伸"），但**它不该出现在 printf 里**——因为 printf 不需要锁。

### 路 3：每进程独立屏幕缓冲 / 真 tty 抽象

**做法**：每个会话有自己的虚拟屏，切换时整屏换。这就是 `tty` + VT 切换 + 终端 tab 模型。

**代价**：极大——多屏幕缓冲、前台切换、scrollback……基本重写终端子系统。

**适用**：Linux / 真 OS。远超现在需要。

---

## 三、为什么选路 1（什么都不改）

### 关键认知：单核 + 单前台 ⇒ 没有并发

两个事实叠起来：

1. **单核**：任意时刻只有一个进程在跑。能打断 printf 的只有**中断**。
2. **单前台输出模型**：同一时刻只有一个进程会 printf（shell 跑着就没别人 print）。

那么 printf 执行到一半时，"另一个进程的 printf 插进来"这个事件**发生的概率是零**——因为没有"另一个进程"在等屏幕。中断里会不会 printf？我们的 ISR（kbd_isr、irq0_enter）都不调 printf，所以也不会。

**没有并发，就不需要互斥。** 加锁是给一个不存在的对手设防。

### "单前台"不是"没 GUI"带来的

这点要分清——我和用户讨论时最初混了：

| | 和 GUI 有关吗 |
|---|---|
| "屏幕会不会乱" | **无关**。纯文本终端，两个进程同时 printf 也会乱。 |
| "谁来决定不乱" | 是**设计选择**，不是 GUI/CLI 决定的。 |

Linux 是 CLI 的，但有成百上千进程，屏幕不乱——不是因为它没 GUI，是因为它有 tty + 前台进程组的输出仲裁。"没有 GUI" ≠ "没有输出竞争问题"。

真正让路 1 成立的，是**"同一时刻只有一个前台输出者"这个约束**。它和 GUI 无关，是一个独立的调度/所有权决策。

### YAGNI：不为想象中的需求提前设计

加锁方案的诱惑在于"将来万一多进程并发 printf 呢"。但：

- 现在 fork 还没做，"多进程并发 printf"是**想象中的需求**。
- 真做 fork 时，第一个撞上的不是"两个进程都 printf"（那要俩都主动输出），而是"子进程的 stdout 归谁"——那是**输入所有权**问题（见下文对偶），和光标锁是两回事。
- 到真有需求时再加锁，**带着真实场景设计**，比现在凭空设计更准。

CLAUDE.md "先理解再动手"在这里落地为：**别为没出现的需求写代码。**

### 约束让简单方案成立

这是这次最值钱的一条元认知：

> **约束没定时，方案是空中楼阁；约束一定，方案自动收敛。**

我和用户讨论时，先抛了三条路让选。用户一句话点破："我根本没打算做 GUI"——他**识别出了一个约束**（单前台模型）。约束一出，路 2/3 立刻作废，路 1 自动胜出。不需要权衡、不需要选，**约束替你选了**。

所以以后遇到"要不要加 X 防御 Y"的问题，第一反应不该是"加不加"，而是"**Y 真的会发生吗？在什么约束下？**"。把约束讲清楚，答案往往自己跳出来。

---

## 四、约束要写下来，不是闷头记住

路 1 的选择依赖"单前台"这个约束。这个约束**必须显式记录**，原因有二：

1. **防止将来自己破约束**。做 fork 时如果忘了"只有一个前台输出者"，让子进程随便 printf，屏幕就乱了——而且到时候会以为是 fork 的 bug，查半天查不到，其实是**输出所有权的约束被破坏了**。
2. **让约束变成可检查的设计原则**。写下来，将来 review 代码时能问一句"这个新进程的输出符合单前台模型吗？"。

所以把这条加进设计原则：**本内核采用单前台模型，同一时刻只允许一个进程向控制台输出。** 将来 fork 时，子进程的 stdout 要么让位给父（父 wait）、要么走别的不抢前台的通道。

---

## 五、对偶问题：输入所有权（fork 时绕不开）

光标竞争是**输出**侧的问题。它有个对偶——**输入所有权**：

> 现在 `kbd_waiter` 是单变量，只能一个进程等键盘。shell 等着没问题。但 fork 出的子进程也调 getchar 会怎样？两个进程都把自己设成 `kbd_waiter`，后一个覆盖前一个，前一个永远醒不来。

输出竞争（多个 printf 抢 cursor）和输入竞争（多个 getchar 抢 kbd_waiter）是**同一枚硬币的两面**：

| | 输出侧 | 输入侧 |
|---|---|---|
| 共享资源 | `console_cursor` | `kbd_waiter` + 键盘 buffer |
| 多进程同时访问 | 字符交错乱码 | waiter 互相覆盖,有人睡死 |
| 单前台约束解决? | 是(只一个 printf) | 是(只一个 getchar) |

**fork 时的解法还是单前台**：同一时刻只有一个进程是 stdin 的前台。fork 时把 stdin 所有权交给子进程，父进程等子退出再拿回。

这就是 Unix `wait()` 的**第二层含义**——你以为 fork+exec+wait 是"管理子进程生命周期"，其实它**同时**解决了"谁拥有 stdin/stdout"。一鱼两吃：`wait` 既回收子进程，又把前台还给父。

所以将来做 fork 时，光标竞争这个"问题"会自然被 `wait` 顺手解决——**根本不需要给 printf 加锁**。这也反过来印证了路 1 的正确性：等真正的需求（fork）来了，解法（wait 的前台语义）会一并覆盖它，现在加的锁到时候是多余的。

---

## 六、延伸：`irq_save`/`irq_restore` 这对原语本身

虽然这次没在 printf 里用，但讨论中引出的 `irq_save`/`irq_restore`（vs 裸 `cli`/`sti`）是一条独立有用的知识，记一下：

### 裸 cli/sti 的坑

```c
void printf(...) {
    cli();
    ... 整行 ...
    sti();    // ← 危险:无条件开中断
}
```

如果 printf 是被一个**中断处理程序**调用的（ISR 里想打印调试信息），此时 IF 本来就是 0（中断进来自动关）。printf 用 `sti` 收尾会**错误地把中断打开**——ISR 还没返回呢，中断被开了，可能出乱子。

### irq_save / irq_restore：保存-恢复 IF

```c
unsigned long flags;
flags = irq_save();    // 关中断,记下"原来 IF 是 1 还是 0"
... 临界区 ...
irq_restore(flags);    // 恢复成原来的 IF
```

- 进来 IF=1 → 关掉、记住"原来是开的"→ 出去 `sti`
- 进来 IF=0（中断里）→ 关掉、记住"原来是关的"→ 出去保持关

**不改变"是否开中断"的整体状态，只借一段关中断的时间。**

### 和 Linux 的对应

这就是 Linux `spin_lock_irqsave` / `spin_unlock_irqrestore` 干的事——名字长就是因为它把"保存中断状态"和"加自旋锁"两件事打包。咱们 UP（单核）不用真锁，只保留"保存/恢复中断状态"那半。

| | 这里(UP) | Linux(SMP) |
|---|---|---|
| 关本核中断 | `cli` | `cli`（在 spin_lock_irqsave 里） |
| 拦别核 | 不需要(单核) | 自旋锁(`spin_lock`) |
| 保存/恢复 IF | `irq_save`/`irq_restore` | 同上 |

**什么时候真要用它**：将来某个临界区可能被**中断 + 进程上下文**两种路径访问，且中断里也调它——这时裸 `cli`/`sti` 就不够了，得上 `irq_save`/`irq_restore`。printf 不是这种场景（没人从中断里调 printf），所以不需要。

## 七、决策记录

| 问题 | printf 要不要加锁防多进程行内交错 |
|---|---|
| 决定 | **不加。什么都不改。** |
| 理由 | 单核 + 单前台输出模型 ⇒ 不存在并发 printf 的场景,锁是多余防御 |
| 前提约束 | 本内核同一时刻只有一个前台输出者(shell 或它授权的子进程) |
| 何时 revisit | 真 fork 时,若出现"子进程要输出"的真实需求,再按当时场景设计(很可能是 wait 的前台语义顺手解决,不需要 printf 锁) |

## 八、术语速查

| 术语 | 含义 |
|------|------|
| 行内交错 | 一个 printf 半句被另一个 printf 劈开,字符穿插(真问题) |
| 行间交错 | A 整行、B 整行交替(可接受) |
| 单前台模型 | 同一时刻只有一个进程拥有控制台输出/输入权 |
| YAGNI | You Aren't Gonna Need It,不为想象中的需求提前设计 |
| irq_save/irq_restore | 保存-恢复 IF 的关中断原语(比裸 cli/sti 安全) |
| 前台进程组 | Unix 里拥有终端 stdin/stdout 的进程组,wait 时切换 |
| 对偶 | 输出竞争(cursor)与输入竞争(kbd_waiter)互为对偶,同源同解 |

## 九、和 Linux 的对应

| 概念 | 这里 | Linux |
|------|------|-------|
| 输出仲裁 | 单前台约定(无代码) | tty 驱动 + 前台进程组 + 终端锁 |
| 输入所有权 | `kbd_waiter` 单变量 | tty 的 `tty->read_wait_queue`,前台进程组读 |
| 前台切换 | (将来)fork 时手动 | `tcsetpgrp()` 改前台进程组 |
| wait 的双重含义 | (将来)回收子 + 拿回前台 | `wait()` 回收 + SIGTTOU/SIGTTIN 前台语义 |
| 关中断原语 | `irq_save`/`irq_restore`(未用) | `spin_lock_irqsave`/`spin_unlock_irqrestore` |

骨架对应,只是 Linux 把"单前台"从约定升级成了由 tty 子系统强制执行的机制。我们靠自觉,Linux 靠硬件+信号。
