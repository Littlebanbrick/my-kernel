## 核心原则

1.  **先理解，再动手**：在写代码之前，必须先确保理解"为什么这么做"。遇到不清楚的概念，先提问、先查阅，不要猜。
2.  **学知识比跑代码更重要**：写对了但环境限制无法验证的情况是存在的。我们的目标是理解内核的原理，而不是执着于让它在某个特定环境下跑通。
3.  **调试优先**：学会用 QEMU + GDB 调试，比写代码本身更重要。"不知道怎么调"是拦路虎，遇到问题先想"我怎么用 GDB 看到底发生了什么"。
4.  **文档即记忆**：重要的概念解释、排查过程、学习心得，整理成 `.md` 文件放在 `agentic/` 目录下。这是"写给未来的自己看的笔记"。
5.  **每步都提交**：每完成一个可运行的小功能就 `git commit`，提交信息用英文，格式参考 Linux 内核社区的 commit message 风格。

## 技术栈与约束

- **主要语言**：C (C99/C11) + x86_64 汇编（AT&T 语法，与 Linux 内核保持一致）
- **构建系统**：Make，使用独立构建目录（`build/`）
- **模拟器**：QEMU（唯一运行环境，不用真机）
- **调试器**：GDB，通过 QEMU 的 `-s -S` 参数远程调试
- **版本控制**：Git，所有变更通过 Git 管理
- **代码风格**：遵循 Linux 内核编码风格
- **代码语言**：纯英文（标识符、注释、字符串均使用英文）
- **文档语言**：中文（`agentic/` 目录下的笔记和解释使用中文）

## 常用命令

- **构建内核**（在对应 stage 目录下运行）：
  ```bash
  make
  ```

- **在 QEMU 中运行**：
  ```bash
  make run
  ```

- **启动 QEMU + GDB 调试**：
  ```bash
  # 终端 1：启动 QEMU（等待 GDB 连接）
  make debug

  # 终端 2：连接 GDB（boot.S）
  gdb build/boot.elf -ex "set architecture i386:x86-64" -ex "target remote localhost:1234"

  # 注：跳转到 stage3 后需加载符号
  # (gdb) add-symbol-file build/stage3.elf 0xB000
  ```

## 工作流程

1. **新会话开始时**：你告诉我目前想做哪个子任务，我们回顾当前进度，规划下一步。
2. **任务分解**：每个功能拆成可调试、可验证的小步骤（例如"在屏幕上打印一个字符"就是一个独立步骤）。
3. **迭代**：每步流程为 计划 → 写代码 → 构建 → 调试运行 → commit → 写文档。
4. **遇到困难时**：先告诉我①你预期该发生什么②实际发生了什么③你尝试过什么。我们一起用 GDB 查看现场。
