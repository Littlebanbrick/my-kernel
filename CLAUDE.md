# CLAUDE.md - 内核开发学习助手配置

## 核心原则

1.  **先阅读，再编码**：在编写任何新代码前，必须先阅读并理解要修改的文件、项目中的类似实现以及相关文档。
2.  **先思考，再动手**：在明确需求、说清假设之前，不要开始写代码。遇到不确定的地方，先提问。
3.  **主动学习与记录**：Claude 像一位阅读速度很快但容易“遗忘”的实习生。在对话中获得的新知识、新发现，**必须**主动总结并更新到此 `CLAUDE.md` 文件中。
4.  **构建与测试驱动**：任何代码变更都必须能够成功构建并通过相关测试。遇到编译或测试失败，优先分析并修复。
5.  **保持对话专注**：对于复杂的、可独立的任务，可以建议使用专门的子代理（Subagent）来处理，以保持主对话上下文的清洁。

## 技术栈与约束

- **主要语言**：C (C99/C11 标准) 和 x86_64/ARM64 汇编 (AT&T 或 Intel 语法，请保持一致)。
- **构建系统**：Make。编译时应使用独立的构建输出目录（如 `claude-build/` 或 `build/`）。
- **模拟/虚拟化**：QEMU。它用于运行和调试内核，是主要的实验环境。
- **调试工具**：GDB。通过 QEMU 的 `-s` 和 `-S` 参数进行远程调试。
- **版本控制**：Git。所有的代码和文档变更都应通过 Git 进行管理。
- **代码风格**：遵循 Linux 内核编码风格 (可参考 `Documentation/process/coding-style.rst`)。
- **文档规范**：所有新增的 `.md` 文档和临时文件，请统一放在 `agentic/` 目录下。

## 常用命令

- **构建内核**：
  ```bash
  make O=claude-build/ -j$(nproc)
  ```
  其中 `O=claude-build/` 指定了构建输出目录，`-j$(nproc)` 用于并行编译以加快速度。

- **在 QEMU 中运行**：
  ```bash
  qemu-system-x86_64 -kernel <path-to-kernel-binary> [其他选项]
  ```

- **在 QEMU 中调试**：
  ```bash
  qemu-system-x86_64 -s -S -kernel <path-to-kernel-binary>
  ```
  然后在另一个终端使用 `gdb` 连接 `localhost:1234`。

- **运行测试** (如果项目已包含)：
  ```bash
  make O=claude-build/ test
  ```
  或参考 `Documentation/` 下的具体测试说明（如 KUnit）。

## 工作流程建议

1.  **新会话初始化**：每次开始新会话时，首先阅读此 `CLAUDE.md` 文件以恢复上下文。
2.  **任务分解**：将复杂任务（如“实现一个简单的内存分配器”）分解为多个可验证的小步骤。
3.  **迭代开发**：对于每一步：
    a.  **计划**：明确要做什么，如何验证。
    b.  **实现**：编写代码。
    c.  **构建与测试**：运行构建命令和测试，确保通过。
    d.  **提交**：编写清晰的提交信息（可参考项目中过往的优秀提交历史），提交代码。
    e.  **记录**：将新学到的知识点或重要的决策记录在此 `CLAUDE.md` 文件的“会话日志”部分。
4.  **遇到困难时**：清晰地描述问题、你尝试过的解决方法以及预期的行为，我会尽力提供帮助。
