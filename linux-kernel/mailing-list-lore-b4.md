# 邮件列表 / lore / b4 —— 内核社区的信息流怎么用

> 为什么内核不用 GitHub PR，而用邮件？怎么"订阅"、到底看什么？本文回答这三个问题。
> 上手顺序：姿势 A（网页归档）→ 姿势 B（`b4` 本地拉）→ 姿势 C（真订邮箱，仅低流量列表）。

---

## 一、为什么内核用邮件，而不是 GitHub PR

这是理解整件事的前提。Linux 内核的协作**几乎完全建立在邮件上**——不是 GitHub、不是 GitLab、不是网页表单。两个原因：

1. **历史与去中心化**：内核开发比 GitHub 早了将近 20 年，沉淀下来的整套工具链（`git send-email`、`git am`、`b4`）和流程都是围绕纯文本邮件设计的。它不依赖任何单一平台，邮件列表由 `vger.kernel.org` 托管，公开、可归档、可镜像。

2. **邮件天然适合 review**：一个补丁就是一封纯文本邮件，reviewer 直接在邮件里逐行回复（quote + 评论），整个讨论线索（thread）可被归档、可被 `git am` 重新应用。GitHub 的 review 绑定在某个 fork 上，而内核的补丁要在 maintainer 的多个子树之间流转，邮件是唯一能解耦的载体。

所以，**"订阅邮件列表" = 加入这个以邮件为媒介的协作网络**，相当于订阅一份每天送到你邮箱的内核开发实时快照。

---

## 二、邮件列表长什么样、流量多大

先建立量级感：

- `linux-kernel@vger.kernel.org`（简称 **LKML**）是总列表，几乎所有补丁都会 Cc 一份到这里。流量是**每天几百到上千封**。直接订到邮箱会让你发疯。
- 子系统列表流量小得多。比如 `rust-for-linux@vger.kernel.org` 单日大概几十封；`linux-sched@vger.kernel.org`（调度器）更少。这些可以直接订到邮箱。

---

## 三、三种"订阅"姿势（从轻到重）

### 姿势 A：不订邮箱，直接看网页归档（推荐新手起步）

**不需要真的让邮件进你邮箱。** 所有内核邮件列表都有公开的网页归档，地址是 **lore.kernel.org**：

- `https://lore.kernel.org/linux-kernel/` — LKML 总列表
- `https://lore.kernel.org/rust-for-linux/` — Rust 子系统

归档实时更新，可按日期、按 thread 浏览。起步阶段用网页看，零成本、零噪音。

### 姿势 B：用 `b4` 工具本地拉取（进阶，强烈推荐）

`b4` 是内核社区现在的主推工具。它不是订阅，而是**按需把某个 thread 从 lore 拉到本地**。在网页上看到一个补丁的 message-id（邮件头里那串 `xxx@yyy.zzz`）后：

```bash
b4 am <message-id>     # 把这个补丁 thread 拉下来，整理成可 git am 的 mbox
```

好处：补丁直接落到本地内核源码树，能干净 apply、能测试、能用 git 工具看 diff。这才是内核开发者真正的工作方式。

### 姿势 C：真的订阅到邮箱（只对低流量列表）

对 `rust-for-linux` 这种低流量列表，可以让邮件真的发到邮箱。给管理地址发一封订阅邮件，例如：

```
To: rust-for-linux+subscribe@vger.kernel.org
Subject: subscribe
```

正文随便写或留空。系统会回一封确认信，回复一下就生效。退订把 `subscribe` 换成 `unsubscribe`。

**LKML 千万别这么订**——会被淹。

## 四、到底要看其中的什么

订阅不是目的，**看什么**才是。按当前阶段，建议盯这几样：

### 1. 看 `[PATCH]` 邮件的格式（学怎么发）

挑几个近期被合并的补丁，看它的主题行、正文结构、`Signed-off-by`、`---` 分隔符、diffstat。这比读任何文档都直观——会看到 `[PATCH 1/3] subsystem: summary` 长什么样，cover letter 怎么写，reviewer 怎么引用回复。**用真实样本校准"补丁长什么样"的心智模型。**

### 2. 看 review 是怎么发生的（学社区文化）

找一个补丁 thread，从头看到尾。重点看：
- reviewer 怎么挑毛病（措辞、角度）；
- 作者怎么回应（接受、解释、还是礼貌反驳）；
- `Reviewed-by:` / `Acked-by:` / `Tested-by:` 这些 tag 怎么被积累和使用——它们就是内核社区的"信任积分"。

会迅速建立直觉：什么样的补丁会被快速接受，什么样的会反复来回。

### 3. 看感兴趣子系统的近期动向（找切入点）

对调度器感兴趣，盯 `linux-sched` 和 `fair.c` 相关 thread；对 Rust 感兴趣，盯 `rust-for-linux`。看 maintainer 们最近在讨论什么痛点、在抱怨什么——**这些就是潜在的切入点**。报告里"开放问题"那几条（Rust 张力、PREEMPT_RT 合入细节、LLM 补丁政策），最权威的答案全在 lore 的历史 thread 里。

### 4. 看 rc 阶段的"回归报告"和"bug 修复"讨论

rc 阶段（如 7.2-rc3）正是稳定化阶段，列表上大量是"这个 patch 引入了 regression / 这个 bug 在某机型上复现"的讨论。这是 [[contributing-roadmap-2026-07]] 里说的"推荐新手入口"——**清 regressions 和 open bugs** 的真实战场。看别人怎么报 bug、怎么给 fix，能直接照着学。

---

## 五、一句话总结

**订阅邮件列表 = 把自己泡在内核开发的真实信息流里，目的不是"收邮件"，而是用 lore 网页归档 + `b4` 工具，去读真实的补丁格式、review 文化、和子系统痛点。** 起步用姿势 A（网页看），上手后转姿势 B（`b4` 本地拉），低流量子系统才用姿势 C（真订邮箱）。
